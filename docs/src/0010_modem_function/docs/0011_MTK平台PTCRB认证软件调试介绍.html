PTCRB认证简介

PTCRB认证简介

PTCRB全称PCS Type Certification Review Board（PCS型号认证委员会），成立于1997年3月，并于2001年5月加入了对GSM850频段的要求，是美国标准化组织发展的一个重要的里程碑。与GCF类似的，PTCRB也是有运营商和一些大的手机厂商组成，还包括一些认可实验室。PTCRB的成员分为两个级别，完全会员（Full or Active Members）和观察员（Passive or Observing Members）。PTCRB成员的主要责任是：1.制定PTCRB认证的要求；2.制定实验室认可的标准；3.通过每年4次会议对标准进行更新；4.发布决议。

PTCRB旗下还有一个重要的团队PVG（PCS Validation Group），他在一年4次的会议中进行讨论，例如：测试设备所遇到的困难；对在特殊环境下操作使用测试设备的统一化等，同时对其标准NAPRD.03进行更新。

PTCRB认证的详细内容及认证流程列于NAPRD.03中，而NAPRD.03-TC列出了参考3GPP标准和OMA标准而制定的测试项。

具体PTCRB测试内容：

1、RF部分的测试；

2、Protocol部分的测试；

3、SIM部分的测试；

4、Audio部分的测试；

认证答疑

Q：为什么要做PTCRB认证?

A：PTCRB是准强制性，强制性应该是政府监管部门发布的要求，但PTCRB不是政府性的，是论坛形式的，却非常具有权威性，其原因在于进入北美的经销商必须按照PTCRB的要求去测试厂家提供的产品，因此也可以理解是强制性。PTCRB同时也是美国各大运营入库认证的基础，比如：AT&T认证、VERIZON认证、T-mobile认证和Sprint认证等，在做以上运营商入库就需要有PTCRB认证的前置条件。

Q：谁需要做PTCRB认证？

A：模块制造商以及终端产品制造商，实现PTCRB认证模块，都有义务遵循PTCRB认证流程。PTCRB认证适用于运营在面向GSM850和190MHz频段、FDD II/IV、UMTS V及LTE的所有PTCRB相关eFDD频段。

Q：如何节省PTCRB认证费用？

A：在终端产品上使用模块，模块厂商会生产一些已做认证的模块，使用已认证模块的模块是最简单直接的办法。在这些情况下，仅需要少量的测试即可。因为当模块获得认证时，大多数测试已经完成了，执行这些后续测试只是确保与集成相关的接口满足PTCRB要求，这样可以大大节约认证费用以及缩短周期。

Q：PTCRB认证会不会出问题过不了？

A：预测试是帮助避免出现意外问题的解决方法，预认证测试包括杂散测试（Radiated Spurious Emissions简称 RSE）及CTIA OTA测试等。预测试会检测产品潜在的问题，可以在这个过程及时进行整改，正式的PTCRB认证测试部分是在国外进行，如果中产生复测将大大提高认证成本。

Q：PTCRB认证怎么进行?

A：PTCRB认证流程由CTIA通过PTCRB网站进行管理；先在PTCRB官网注册注册账号；接收到PTCRB发送到注册邮箱的用户名及密码，访问PTCRB认证数据库；登记公司名字和产品信息，发起PTCRB认证请求，指定测试实验室，接收请求号码；通知实验室通过接收PTCRB官网发送的认证测试请求；实验室通过在线数据库接受了请求，就可以开始测试、测试整改、测试和场测完成；认证管理费用，CTIA将会开具“认证管理费用”发票，金额取决于产品技术及认证类型。通过邮件收到来自PTCRB的认证批准，同时认证审批也显示在PTCRB网站上。 通信产品在北美认证有三个阶段，分别是最基础的FCC ID认证、常规的运营商的PTCRB认证、以及比较高端的各大运营商认证。各大运营商为确保进网通信产品的一致性和提高产品的网络实际兼容性的测试要求。值得一提的是这些要求成为了国内各大运营商入库要求的蓝本，所以我们通常也会习惯的将各大运营商认证称之为运营商入库认证，与PTCRB区分开来。

北美PTCRB测试要求

在美国，运营商准入认证又称PTCRB认证测试。PTCRB是1997年在北美成立的一个负责PCS型号认证的审核委员会，同时成立的还有一个验证组，主要负责PCR移动终端型号的认证和IMEI分配。

​ PTCRB认证是第三方认证机构执行的准强制型号认证，所有投放北美市场的PCS终端设备都要经过PTCRB认证，并依据报告申请IMEI。这里用的是准强制性，强制性应该是政府监管部门发布的要求，但PTCRB不是政府性的，是论坛形式的，但是非常具有权威性，其原因在于进入北美的经销商必须按照这个要求去测试厂家提供的产品，因此叫做准强制性。

PTCRB测试周期

测试周期主要看手机支持的功能而定。比如手机支持的频段、手机是否支持GPRS Class10、是否支持EDGE、是否支持半速率等等，根据不同的情况，测试的时间长短也相差很大，而且测试周期还收到测试过程手机整改和调试的影响。

​ 一般来讲一个PTCRB测试的全型如果测试顺利，测试周期一般需要4周左右的时间。

3.其他注意事项

[确认硬件篇]

首先确认Sample RF都做了==准确的校准==，有数量足够（LTE项目至少需提供十把手机， 实际数量请跟lab讨论）且RF 和天线性能优异的手机给LAB随时准备用于复测。

– 测 Acoustic 的手机有调过曲线以及参数测试仪器都确定设置准确

[手机配置篇]

(important请务必在测试前确认这些设置，务必同时提醒到Lab!!)

首先确认UE和仪器屏蔽良好并连接正常。对于RF，RR，RRC相关case与Sample的射频性能也有一定关系，建议可以使用不同的sample进行复测来避免硬件的个体差异。测试/复测前，确认device设置正确。

  • 1.打开FTA mode:工程模式->modem test->选择FTA，进入后选择测试的Test Site（Anite, CRTUG, CRTUW, ANRITSU, CMW500），重启后再次进入设定界面确认设置是否成功。

（比如，选择了Anite，The CURRENT MODE IS FTA: Anite）。Ps.如果不清楚Test Site，可以把所有test site都勾上。

  • 2.打开data prefer: 工程模式->Mobile data service preferred->勾选上即可；

  • 3.Quick Boot

Settings -> Accessibility -> Quick boot ->取消勾选，有测试都关掉, 避免遇到不预期的error.

  • 4.背景流量禁用:

在手机菜单：关闭背景数据

Settings -> Data Usage-> Data Saver菜单 选择ON

– Engineering mode->Background Data Select ->Disable Background Data Select

[ES提交篇]

  • 按SOP，在第一轮测试完成之后进行复测，复测完成之后可以开始提交FTA issue CR. 在第一轮测试完成 or 第一轮复测完, 如果没有MTK log, 请勿提交e-Service. 其中,fail case ID请严格copy Lab report statistics中的fail case ID，以免产生误解；

  • 将fail case对应的所需log附上eservice ，并且把fail的description写的详细一点；

  • CR Title规则例如：“[project_name] [GCF] test case specification + fail case id + fail case title ”

    • 举例：“[DD340][GCF] 31.124_27.22.4.26.2/3 _LAUNCH BROWSER (Interaction with current session) ”

    • 举例：[G1925PR][PTCRB][ETC__IMS][3GPP TS 34.229-1][G.15.17][Subscription to the MWI event package / WLAN]

[Log要求篇]

第一轮测试后无需提供Fail Case Log，第一轮测试的目的是尽快过完所有Case，第一轮测试后提供所有Case状态表即可；

第一轮复测完毕后,仍然Fail的Case安排Log抓取，注意一条Fail Case对应一份Log而且是从开机开始录制的log，所谓抓开机log；

所有复测阶段fail的case，请保存相应的Machine Log与MTK (AP+Modm) Log，并提交到eservice，目前已经提交eservice的fail case，请帮忙提供相应的Machine Log与MTK (AP+Modem) Log

Log抓取要求：

 1.针对3G protocol fail case，请进行retest，抓取Mtklog（mobile log + modem log）及仪器log。

 2.针对(U)SIM/(U)STK/UICC fail case, 请进行retest，抓取Mtklog（mobile log + modem log +Netlog）、仪器log及IT3 报告。

 3.针对HW fail case，请抓取Mtklog（mobile log + modem log）及仪器log；

注意：仪表log请让Lab转换成HTML格式。

– 如何抓开机log:

– 请到工程模式->MTK Logger

– 进入MTK Logger选项后右上角setting按钮，

– 分别点击Mobile Log/Modem Log/Net Log -> 选中 ”Automatically record when reboot”or “start Automatically”

– 建议Lab行为：(important!!)

– 建议请lab对于protocol等系列fail case，留下comment，指出他们在仪器端操作的时候感观上fail点在哪，以便定位问题；
